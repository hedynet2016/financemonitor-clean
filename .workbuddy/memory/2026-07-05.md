# 2026-07-05 工作紀錄

## 問題排查與修復：經濟指標新聞推播時區錯誤

### 問題描述
用戶在下午 4 點（16:00）收到經濟指標相關新聞推播，但預期應在早上 8 點（08:00）發送。此外，新聞標題為英文，需翻譯為中文。

### 根因分析
1. **時區問題**：`datetime.now()` 使用系統本地時間，但 Render 的 `python:3.11-slim` 映像檔預設未安裝 `tzdata`，導致 `TZ=Asia/Taipei` 環境變數無效，系統回退為 UTC 時間。
   - 08:00 UTC = 16:00 台北時間（下午 4 點）
   
2. **重複推播**：`render_scheduler.py` 排程了 `economic_news_push.py`，與 `integrated_monitor.py` 的區塊⑩重複。

3. **新聞標題未翻譯**：`news_monitor.py` 的 `_format_economic_section` 方法直接使用英文標題，未呼叫翻譯功能。

### 修復內容

#### 1. Dockerfile - 安裝 tzdata
- 添加 `tzdata` 套件安裝
- 明確設定時區符號連結

#### 2. render_scheduler.py - 使用明確時區
- 添加 `import pytz`
- 定義 `TAIPEI_TZ = pytz.timezone('Asia/Taipei')`
- 修改排程：`schedule.every().day.at("09:00", TAIPEI_TZ)` 和 `schedule.every().day.at("18:00", TAIPEI_TZ)`
- **移除 `run_economic_news_push`**（避免與 integrated_monitor.py 區塊⑩重複推播）

#### 3. integrated_monitor.py - 所有 worker 使用明確時區
- 添加 `import pytz`
- 在模組層級定義 `TAIPEI_TZ = pytz.timezone('Asia/Taipei')`
- 修改所有 worker 函數：
  - `stock_monitor_worker`（第 213 行）
  - `news_monitor_worker`（第 283 行）
  - `events_monitor_worker`（第 373 行）
  - `product_monitor_worker`（第 545 行）
  - 將 `datetime.now()` 改為 `datetime.now(TAIPEI_TZ)`

#### 4. news_monitor.py - 新聞標題翻譯為中文
- 修改 `_format_economic_section` 方法（第 992 行）
- 添加標題翻譯：`title_zh = self.translate_text(title)`
- 添加摘要翻譯：`summary_zh = self.translate_text(summary[:200])`

#### 5. webui.py - 更新時間顯示
- 將所有 "ET"（美東時間）改為 "台北時間"
- 更新排程模式說明
- 修改區塊⑩描述：移除 "BLS"，改為 "CNBC / WSJ / Bloomberg / MarketWatch 等財經媒體 RSS"

#### 6. requirements.txt - 添加依賴
- 添加 `pytz>=2023.3`

### 後續注意事項
1. **Render 部署**：推送後 Render 會自動重新部署，屆時 Dockerfile 的 `tzdata` 安裝會生效。
2. **驗證推播時間**：明天（2026-07-06）早上 08:00 台北時間應收到經濟指標新聞推播。
3. **驗證中文標題**：推播內容的新聞標題應為中文翻譯結果。

### 技術筆記
- `python:3.11-slim` 映像檔預設不包含 `tzdata`，需手動安裝
- `schedule` 函式庫的 `at()` 方法支援 `pytz` 時區物件
- `datetime.now()` 不支援時區參數，需使用 `datetime.now(pytz_timezone)`
- 模組層級定義時區常數可避免重複程式碼

## 問題排查與修復（續）：notification_sender config key 路徑錯誤（16:18）

### 問題
用戶反映仍然無法收到來自 Render 的 Telegram 推播。

### 根因分析
發現 `notification_sender.py` 有兩個嚴重 bug：

#### Bug 1：config key 路徑不匹配
- `notification_sender.py` 使用 `self.config.get("TELEGRAM_BOT_TOKEN")`（頂層 key）
- 但 `generate_config.py` 生成的 `config.json` 結構為 `config["telegram"]["bot_token"]`
- 結果：`NotificationSender.__init__` 總是 fallback 到 `os.environ`，完全忽略 config.json

#### Bug 2：`send_telegram_message()` 不接受參數
- 獨立函式 `send_telegram_message()` 只讀 `os.environ`，不接受 token/chat_id 參數
- 即使 `NotificationSender` 實例有正確的 token，也無法傳遞給這個函式

### 修復內容

#### notification_sender.py（核心修復）
- `__init__`: 修正為 `tg.get("bot_token")` 和 `dc.get("webhook_url")`（從嵌套結構讀取）
- `send_telegram()`: 傳遞 `self.telegram_token` / `self.telegram_chat_id` 給 `send_telegram_message()`
- `send_discord()`: 傳遞 `self.discord_webhook` 給 `send_discord_message()`
- `send_telegram_message()`: 新增 `bot_token` / `chat_id` 參數，fallback 到 `os.environ`
- `send_notification()`: 新增 `bot_token` / `chat_id` / `webhook_url` 參數並傳遞
- 新增初始化日誌：顯示 telegram/discord/email 配置狀態

#### integrated_monitor.py（診斷增強）
- `news_monitor_worker`: 添加 `try/except` 包裹 `run_news_only()` 以捕獲錯誤
- `news_monitor_worker`: 添加 [HEARTBEAT] 每 30 分鐘輸出存活日誌
- 啟動時輸出配置資訊

#### webui.py（診斷增強）
- 啟動時輸出 [DIAG] 環境變數狀態

#### render_start.sh（診斷增強）
- 啟動時輸出時間、時區、環境變數狀態
